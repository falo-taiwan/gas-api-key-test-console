一、頁面定位

本節先講清楚這個專案到底是做什麼的，以及為什麼我們刻意把內容拆成「說明頁」和「執行頁」。如果你是第一次接觸 Google Apps Script、Google Sheet 與 AI API，先把這一頁看懂，後面操作時會輕鬆很多。

這章在學什麼：理解專案定位、教材頁與操作頁的分工。

實務上拿來做什麼：讓你可以把這個專案同時當成教材、展示頁與測試台。

哪裡容易踩雷：初學者最常把說明、設定、輸入表單與除錯資訊塞在同一頁，最後畫面很滿，但教學效果反而不好。

說明頁

負責架構、規則、角色分流、資料表結構、初始化流程與安全說明。這一頁偏向教材、導讀與部署文件。

執行頁

負責登入、模型切換、送出測試、覆蓋參數、讀取範例網址、驗證管理者密碼，以及觀察結果與 debug 回應。

一句話理解這個專案：它不是正式產品聊天台，而是一個專門拿來教「前端表單怎麼送到 GAS、GAS 怎麼接 Sheet、Sheet 怎麼控制回覆、回覆怎麼再送回前端」的教學型最小系統。

二、v1.02 版本重點

本節把目前版本最重要的設計決策一次整理給你。若你要帶學員看「專案是怎麼一步步長大的」，可以先從這一段開始講。

這章在學什麼：理解這一版和早期單純測試版相比，多了哪些真正可教學的模組。

實務上拿來做什麼：讓你在課堂上快速說明為什麼這不是只有 API 測試，而是有角色、限制、設定來源與安全邊界的系統。

哪裡容易踩雷：只看到前端畫面，卻忽略了後端角色分流、A/B 驗證、Sheet 設定和 audit 計數器其實才是這版的核心。

  • Use Server Key 才會套用角色分流；admin 可拿完整教學版回覆，user 在 Server Key 模式下只進入受限教學回覆。
  • Use My API Key 不受角色限制，改由使用者自己的 key 直接走完整回覆。
  • 前端新增 A / B 驗證機制：一組驗證碼用來取得範例網址，另一組驗證碼用來讀取目前 admin 密碼。
  • 後端支援 runtime_config 與 forced_output，也就是可以從 Google Sheet 直接控制前言、結語、system prompt 與字數限制。
  • 系統新增 AUDIT_TOTAL_EXECUTIONS 稽核計數器，每次執行 doGet / doPost 都會累加。
  • 字數限制預設改為 500，若前端填 0，代表不限制字數。

三、快速部署

本節是給完全沒碰過 Google Apps Script 的新手。你不需要先懂全部技術名詞，只要先照著步驟把環境搭起來，就能在最短時間內看到「前端送出訊息、後端回應結果」這個完整迴路。

這章在學什麼：從 Excel、Google Sheet、Apps Script 到 Web App 的最小部署流程。

實務上拿來做什麼：讓學員或客戶在 10 到 20 分鐘內完成第一個可操作版本。

哪裡容易踩雷：不知道 GAS 在哪裡開、沒有先跑初始化、Script Properties 名稱打錯、部署後拿錯網址。

1
上傳 Excel 並轉成 Google Sheet
把 Api-Key-Teach-Template.xlsx 上傳到 Google Drive，開啟後轉成 Google 試算表。這份表不是單純資料表，而是整個教學版後端的資料中心，後續帳號、前言、結語、runtime 設定與 logs 都在這裡。
2
從 Google Sheet 開啟 Apps Script
進入「擴充功能 > Apps Script」，把 SheetSetup.gs 與 Code.gs 的內容貼進去。這一步的意思是：讓後端程式和這份 Google Sheet 綁在一起，後端之後就能直接讀這份表裡的帳號、設定與文字。
3
先執行初始化
執行一次 setupTeachingSheets()。它會檢查 README / users / logs / forced_output / runtime_config 是否完整；若結構壞掉，會重建成可教學的預設狀態。這一步可以把它理解成「把後端資料中心整理成程式看得懂的格式」。
4
設定 Script Properties API Key
到「專案設定 > 指令碼屬性」，填入你要用的模型金鑰。沒有填的模型，在前端與後端都會被明確擋下。對新手來說，可以先只填 GEMINI_API_KEY，確定整體可跑再補其他模型。
5
部署成 Web App
新增部署作業，類型選「網頁應用程式」，執行身分選自己，存取權限依你的教學情境決定。部署後會得到一個 /exec 網址，這就是前端真正會打到的後端入口。

你要貼哪些檔案到 GAS？

SheetSetup.gs 負責初始化與修復 Sheet 結構。

Code.gs 負責接收前端請求、驗證登入、讀取設定、呼叫模型與回傳結果。

最快驗證成功的方法

先只填 GEMINI_API_KEY，部署完成後到執行頁用管理者示範流程測試；若已啟用第二組驗證碼，就用它回填 users sheet 中目前的 admin 密碼。

等基本迴路通了，再補 KIMI_API_KEY 與 MINIMAX_API_KEY。這樣你在排錯時比較不會一次混進太多變數。

四、整體架構

本節用系統視角說明這個教學包的模組邊界，幫助新手理解「前端」「後端」「資料表」「模型 API」之間到底怎麼接起來。很多初學者一開始會把 AI 想成只有一個聊天框，但實際上它背後至少有四層。

核心架構：前端執行頁送出 payload，GAS Code.gs 處理請求，Google Sheet 提供帳號與設定，最後再由後端決定是否呼叫 Gemini / Kimi / MiniMax。

  • 前端：收集使用者輸入，送出 message / aiModel / username / password / systemPrompt / outputCharLimit / keySource / userApiKey
  • GAS：驗證使用者、判斷角色、檢查環境變數、讀取後端即時設定、依角色決定回覆模式、呼叫模型、寫入 logs
  • Google Sheet：作為教學版後端資料來源與可視化控制台，提供帳號、前言、結語、動態設定與紀錄
  • 模型 API：依 aiModel 轉向 Gemini / Kimi / MiniMax，不同模型有不同端點與限制

給新手的直覺理解：前端像「表單與畫面」，GAS 像「中控台」，Google Sheet 像「可視化資料庫」，模型 API 才是真正產生 AI 回覆的地方。把這四層分開想，後面很多概念就會自然很多。

五、支援模型

本節整理目前專案實際支援的模型。對教學來說，這一章很重要，因為學生需要先知道「多模型」不是抽象名詞，而是真的要面對不同供應商、不同端點、不同 key、不同限制。

這章在學什麼：理解 Router 為什麼存在，以及不同模型供應商之間不會完全相同。

實務上拿來做什麼：讓學生知道前端雖然只是一個下拉選單，後端其實會因此分流到不同 API。

哪裡容易踩雷：以為所有模型都能用同一種端點、同一種參數、同一種回覆格式。

Gemini

目前教學版主要以 Google Gemini 為預設模型。好處是相對穩定，也適合先建立第一條可用迴路。

Kimi

Moonshot 系列模型。重點不是只有模型本身，而是國際版與中國版端點差異、相容格式與參數限制。

MiniMax

MiniMax M2.7。實務上常遇到 Token Plan 與模型綁定限制，因此很適合拿來教供應商差異。

Router 的價值

前端只要傳一個 aiModel，後端就能根據模型名稱選 API Key、端點與模型名稱，這就是 Router 的核心價值。

六、三大踩坑與解法

本節是這個專案非常值得保留的教學內容。因為初學者常以為「API Key 對了就會通」，但多模型整合真正麻煩的，往往是端點、套餐限制與輸出格式這些隱性問題。

坑 1：Kimi 端點混淆

國際版 API Key 如果誤打到中國版網域，就可能出現 401 Invalid Authentication。重點不是 key 錯，而是網域錯。

解法：國際版請走 https://api.moonshot.ai/v1/chat/completions。

坑 2：MiniMax Token Plan 綁模型

有些 MiniMax 金鑰不是「任何模型都能打」，而是綁定某個 Token Plan 與模型。你可能會遇到 key 看起來沒問題，但實際上被套餐限制擋下。

解法：固定對應端點與模型，例如教學版使用 MiniMax-M2.7。

坑 3：CoT / think 汙染 JSON

某些模型會在 JSON 前多吐出 <think> 或其他思考鏈內容，導致後端 JSON.parse() 失敗。

解法：先清掉思考鏈，再抽出真正的 JSON 主體。

教學角度怎麼講

這三個坑很適合拿來教「程式錯誤不一定是你程式碼壞掉，也可能是供應商規格、環境設定或回傳格式不一致」。

七、Google Sheet 結構

本節整理這個教學包目前使用的 5 張工作表，以及它們各自扮演的角色。對新手來說，這一章非常重要，因為它會幫你看懂：這份 Google Sheet 並不是附屬品，而是系統的一部分。

這章在學什麼：把資料表當成後端設定的一部分，而不只是靜態表格。

實務上拿來做什麼：讓學生可以不改程式，就先透過 Sheet 理解帳號、設定與輸出控制。

哪裡容易踩雷：表頭被改掉、欄位順序跑掉，導致初始化或讀取失敗；或以為只改 Sheet 名稱沒差，實際上後端會直接找不到。

  • README：操作說明、部署方式、教學規則與環境變數提醒。它像是放在 Sheet 裡的內建使用手冊。
  • users：帳號密碼與角色，預設 admin / 123、user / 123。角色限制只在 Use Server Key 下生效；管理者密碼回填也會讀這張表。
  • logs：使用者輸入、系統輸出與 debug 紀錄。這張表對教學很有價值，因為你可以直接回頭看每次送出了什麼、拿回了什麼。
  • forced_output：前言與結語。它讓你能示範「AI 回覆不是直接裸輸出，而是可以再加上固定的系統文字」。
  • runtime_config：後端即時 systemPrompt 與 outputCharLimit。它讓你在不改程式的前提下，直接從 Sheet 調整回覆風格與限制值。

給新手的直覺理解：users 像登入資料表，runtime_config 像系統設定表，forced_output 像固定片頭片尾，logs 像稽核紀錄，README 則像這個系統內建的說明書。

八、後端即時設定

本節說明目前最重要的新設計：系統限制不再全部寫死在程式裡，而是優先從 Google Sheet 讀取。這對教學非常重要，因為你可以讓學生不用改一行 JavaScript，就先體會「後端設定改了，前端結果就會變」。

即時控制邏輯：先讀 runtime_config，若前端 payload 有傳同名欄位，再以前端值覆蓋。也就是說，系統同時存在「後端預設值」與「前端臨時覆蓋值」兩個層次。

system_prompt

預設值是教學導向、易讀、偏精簡的回覆風格。它控制的是「AI 會用什麼語氣與方式回答」。

output_char_limit

預設值：500。回覆內容會在後端被真正截斷，不只是提示文字；若填入 0，代表不限制字數。

教學角度怎麼講：你可以讓學生先把前端的 System Prompt 覆蓋值 清空，觀察它改讀 runtime_config；再改填前端值，觀察這次改由前端覆蓋。這樣很容易理解什麼叫做多層設定來源。

九、環境變數與安全邊界

本節專門講「哪些東西應該放在 Script Properties、哪些資訊不能直接寫死在公開版前端，以及目前這個專案怎麼做 A / B 驗證」。這章對新手很重要，因為很多人會先把系統做出來，最後才想到安全性。

這章在學什麼：區分模型金鑰、分享驗證碼、live exec 入口與角色密碼這幾種不同性質的資訊。

實務上拿來做什麼：讓專案可以教學、示範、分享，但又不要把真正危險的入口直接裸露出去。

哪裡容易踩雷：把真實 exec 網址、正式帳密或 server key 直接寫死在 HTML 裡，之後只要頁面外流，就等於把入口一起送出去。

主要 AI 金鑰

GEMINI_API_KEY、KIMI_API_KEY、MINIMAX_API_KEY 都應放在 Script Properties，而不是前端。

A 機制：取得範例網址

SHARE_EXEC_CODE 驗證通過後，後端才會回傳 SHARE_EXEC_URL。這是受控分享範例網址的設計。

B 機制：載入管理者密碼

ADMIN_PASSWORD_CODE 驗證通過後，後端才會去讀 users sheet 中 admin 的目前密碼並回填。

稽核計數器

AUDIT_TOTAL_EXECUTIONS 會在每次執行 doGet / doPost 時累加，右上角會顯示成 Audit 119 這種格式。

公開版安全備註：若這份專案要放到 GitHub 或提供學員下載，真正較高風險的是可直接呼叫後端的 GAS Web App /exec 網址。這份專案主檔已刻意不內建真實網址；建議改用手動貼上、受控發放，或另外建立專用測試端點。

十、執行流程

本節用步驟方式說明使用者從按下送出，到看到回覆結果，中間系統做了哪些事。這一段很適合你後面再補流程圖，因為現在文字結構已經先整理成清楚的步驟。

  1. 前端執行頁收集使用者輸入與測試參數。
  2. GAS Code.gs 解析 payload，決定模型與 key 狀態，並先確認目前選到的模型在環境變數裡是否有對應金鑰。
  3. 若有帳號密碼，就到 users 檢查角色、密碼與啟用狀態。
  4. 若目前是 Use Server Key，才套用角色分流：admin 回傳完整教學版回覆，user 進入受限教學回覆。
  5. 若目前是 Use My API Key，則不套用 admin / user 限制，直接走完整回覆。
  6. 從 runtime_config 讀取後端限制，必要時再被前端覆蓋。
  7. 從 forced_output 取出前言與結語，並組裝成最後的混合輸出。
  8. 若走完整模型模式，就依 aiModel 分流到 Gemini / Kimi / MiniMax。
  9. 後端把結果、debug 資訊與 replySections 一起回傳給前端。
  10. 前端把回覆整理成可教學的顯示格式：前言、AI 回覆、後語，以及額外的系統摘要。

十一、教學建議

本節提供實際帶課時的使用建議，幫助你把這個專案轉成可操作、可展示、可拆解的教材。因為這個專案不只是能跑，而是真的很適合拿來講「AI 系統是怎麼被組起來的」。

再帶學生打開執行頁，實際送出訊息與觀察 debug 回傳，特別是右上角的 System Summary。

十二、結尾總結
